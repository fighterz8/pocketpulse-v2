## Part 2 — Current database state

Run these four SQL queries against the production database. Paste each result verbatim. Identify the target user as the most recently active `is_dev = true` user.

### Query A — labelSource distribution

```sql
SELECT label_source, COUNT(*) 
FROM transactions 
WHERE user_id = <dev user id>
  AND user_corrected = false 
  AND excluded_from_analysis = false
GROUP BY label_source
ORDER BY COUNT(*) DESC;
```

### Query B — confidence distribution within `rule` rows

This reveals how many "rule" rows are actually fallback rows with confidence 0.55 (the no-match signature).

```sql
SELECT 
  ROUND(label_confidence, 2) AS confidence_bucket,
  COUNT(*) 
FROM transactions 
WHERE user_id = <dev user id>
  AND user_corrected = false 
  AND excluded_from_analysis = false
  AND label_source = 'rule'
GROUP BY ROUND(label_confidence, 2)
ORDER BY confidence_bucket;
```

### Query C — the `aiAssisted` flag independently of labelSource

A row can have `aiAssisted = true` (pipeline marked it as a candidate for AI) without having `labelSource = 'ai'` (AI didn't actually overwrite it). This query shows the discrepancy if one exists.

```sql
SELECT 
  label_source,
  ai_assisted,
  COUNT(*)
FROM transactions 
WHERE user_id = <dev user id>
  AND user_corrected = false 
  AND excluded_from_analysis = false
GROUP BY label_source, ai_assisted
ORDER BY label_source, ai_assisted;
```

### Query D — recency check (is AI firing on recent uploads specifically?)

```sql
SELECT 
  DATE(created_at) AS upload_date,
  label_source,
  COUNT(*)
FROM transactions
WHERE user_id = <dev user id>
  AND user_corrected = false
  AND excluded_from_analysis = false
  AND created_at > NOW() - INTERVAL '7 days'
GROUP BY DATE(created_at), label_source
ORDER BY upload_date DESC, label_source;
```
