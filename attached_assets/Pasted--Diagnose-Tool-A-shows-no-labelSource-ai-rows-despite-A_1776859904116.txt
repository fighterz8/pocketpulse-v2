## Context

- Yet every run of the Dev Test Suite's Tool A (`/dev/test-suite/classification`) returns samples where every row's `labelSource` is `"rule"` or `"cache"`. Never `"ai"`.

## Check 1 — Direct DB query: is `labelSource = 'ai'` actually in the data?

This is the fastest check and potentially the most decisive. Run it first because everything else depends on whether AI-labeled rows exist in the DB at all.

### What to do

Run these two SQL queries against the production database. Use the most recently active dev user's user ID — identify that user by selecting from `users` where `is_dev = true` ordered by most recent activity (or ask the user for their ID if ambiguous):

```sql
-- Query 1: labelSource distribution for this user's eligible transactions
SELECT label_source, COUNT(*) 
FROM transactions 
WHERE user_id = <dev user id>
  AND user_corrected = false 
  AND excluded_from_analysis = false
GROUP BY label_source
ORDER BY COUNT(*) DESC;
```

```sql
-- Query 2: the AI-labeled row pool as a percentage of the sampling pool
-- (the sampling pool is what Tool A's random sampler draws from)
SELECT 
  COUNT(*) FILTER (WHERE label_source = 'ai') AS ai_count,
  COUNT(*) FILTER (WHERE label_source IN ('rule', 'cache', 'ai', 'user-rule')) AS eligible_pool,
  ROUND(
    100.0 * COUNT(*) FILTER (WHERE label_source = 'ai') / 
    NULLIF(COUNT(*) FILTER (WHERE label_source IN ('rule', 'cache', 'ai', 'user-rule')), 0),
    2
  ) AS ai_percent_of_pool
FROM transactions 
WHERE user_id = <dev user id>
  AND user_corrected = false 
  AND excluded_from_analysis = false;
```

### What to report

Paste both query results verbatim, then interpret:

- **If Query 1 shows zero rows with `label_source = 'ai'`**: the AI-firing seen in the OpenAI dashboard is not landing in the DB as `'ai'`. Something in the persistence path is overwriting `labelSource`. This is a persistence bug. Describe the likely cause in one paragraph (the prime suspect is the cache writeback block in `classifyPipeline.ts` — look at whether AI results get re-read from cache after being written, which would demote `'ai'` to `'cache'`). Do not fix. Stop.

- **If Query 1 shows rows with `label_source = 'ai'` AND Query 2 shows `ai_percent_of_pool` is ≥ 5%**: AI rows exist and are a reasonable fraction of the pool. The problem is in Tool A's sampling query — it's filtering AI rows out. Proceed to Check 3 (in a new session).

- **If Query 1 shows rows with `label_source = 'ai'` AND Query 2 shows `ai_percent_of_pool` is < 5%**: AI rows exist but are statistically rare in the pool. A 50-row random sample legitimately has a low probability of including any. This is not a bug; it is a sampling design issue. Recommend stratified sampling as the fix. Do not fix. Stop.
